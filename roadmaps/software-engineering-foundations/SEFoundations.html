Dev Environment & Git

Beginner~1 week· Prereq: none
▸
Prerequisites

None. A computer and consistent study time.

What to learn

Set up a professional, reproducible development environment: VS Code, Python via pyenv, virtual environments, pip, the terminal (bash/zsh), and Git/GitHub including SSH keys, branches, and pull requests. Build a learning log habit from day one.

Why it matters

Broken environments and lost work are the most common early blockers. Clean setup and version control are the infrastructure every later phase depends on — without them you'll keep losing hours to avoidable problems.

Subtopics
VS Code config & extensionspyenv / Python installvenv & pipTerminal (bash/zsh)Git init / add / commitBranches & mergesGitHub repos & SSHPull requestsMarkdown READMEs.gitignore
Mini-projects
  • Create a GitHub repo, write a README in Markdown, and push a first commit.
  • Set up a virtual environment and pin dependencies in requirements.txt.
  • Practice branching: create a feature branch, commit to it, and merge via a PR.
Production checkpoint
Production noteEvery project gets its own isolated virtual environment and a committed .gitignore. Committing secrets or __pycache__ to Git is a defect — not a minor mistake.
Common mistakes
AvoidInstalling everything globally; never using branches; committing API keys or large files; over-engineering the tooling setup instead of learning.
Important notes
NoteTool-tweaking is a common form of procrastination. Pick one editor and one env tool, configure the basics, and move on.
Mastery criteria

You can create an isolated environment, push branched code to GitHub with a clean .gitignore, and open a pull request — all without looking up the steps.

Python Mastery

Beginner~4–6 weeks· Prereq: Phase 0
▸
Prerequisites

Phase 0 environment and Git.

What to learn

Idiomatic Python end to end: data types, control flow, functions, classes, modules, error handling, type hints, generators, decorators, context managers, virtual environments, testing with pytest, and async I/O. This is the single most critical foundation for an AI Engineer and can't be skipped or rushed.

Why it matters

Nearly all AI tooling is Python-first. The difference between someone who pastes snippets and someone who ships systems is deep, testable, well-structured Python. Every later phase (data, ML, backend, RAG) depends on fluent Python.

Subtopics
Lists / dicts / sets / tuplesComprehensionsFunctions & *args/**kwargsClasses & dunder methodsModules & packagesExceptions & context managersType hints + mypyGenerators & iteratorsDecoratorsasync / awaitpytest & fixturesrequests / httpxpydantic v2pyproject.toml
Mini-projects
  • Build a CLI tool (e.g. a file organizer or web-API client) with argument parsing and tests.
  • Write a small library with type hints, packaged with pyproject.toml and a full pytest suite.
  • Fetch and validate data from a public API using httpx + pydantic with typed models.
Production checkpoint
Production noteType hints + pydantic validation catch entire classes of bugs before runtime. Treat untyped, untested Python as a liability in any service you intend to deploy.
Common mistakes
AvoidJumping to libraries before you can write a clean class or a test. Shallow Python is the #1 reason later phases feel impossible. Don't skip pytest — it'll save you hours in every project.
Important notes
NoteLearn async early — LLM streaming (AI Phase 12) and concurrent API calls are walls for engineers who never practiced async/await. You don't need deep concurrency theory yet, but the syntax must be muscle memory.
Mastery criteria

You can design a small package with classes, type hints, and tests, handle errors gracefully, call APIs both synchronously and asynchronously, and run mypy . with zero errors.

Data Structures & Algorithms

Beginner~3–5 weeks· Prereq: Phase 1
▸
Prerequisites

Comfortable Python (Phase 1) — you'll implement everything in Python.

What to learn

How computers store and process data efficiently: core data structures (arrays, hash maps, linked lists, trees, graphs, heaps), fundamental algorithms (binary search, BFS/DFS, sorting), and Big-O analysis. Focus on the structures and patterns that appear in real AI/backend engineering work, not competitive programming arcana.

Why it matters

In real systems, the difference between O(n) and O(n²) is the difference between a 200ms job and a 20-minute one. Without this intuition you write code that doesn't scale and you can't debug performance problems in AI pipelines.

Subtopics
Arrays & dynamic arraysHash maps & setsLinked listsStacks & queues (deque)Binary trees & BSTHeaps / priority queuesGraphs (adjacency list)BFS / DFSBinary searchSorting (merge, quick)Recursion & base casesBig-O time & space
Mini-projects
  • Implement a hash map (open addressing) and a binary search from scratch; analyze their Big-O.
  • Solve a real caching problem using a doubly-linked-list + hash map (LRU cache).
  • Build a task scheduler with a priority queue (heapq); profile it under load.
Production checkpoint
Production notePython's built-in dict is a hash map. collections.deque is a doubly-linked list. heapq is a min-heap. Knowing the underlying structure is what lets you pick the right one — and spot when a colleague's code will be slow.
Common mistakes
AvoidGrinding 200+ LeetCode problems without understanding the patterns. You need working fluency for engineering decisions, not contest mastery. Focus on hash maps, trees, queues, and graphs — these appear in real AI engineering code.
Important notes
Skip for nowAdvanced DP (knapsack variants, bitmask DP), complex graph algorithms (Dijkstra variants, flow), and competitive-programming tricks — those are for interview prep at Phase 11+, not here.
Mastery criteria

You can state the Big-O of common operations, implement a hash map and BFS from scratch, and explain why an LRU cache uses a doubly-linked list + hash map.

Trusted resources
MIT 6.006 Intro to Algorithms (OCW)ocw.mit.edu Python collections moduledocs.python.org Python heapqdocs.python.org VisuAlgo — Algorithm Visualizationsvisualgo.net

Operating Systems & Networking Fundamentals

Intermediate~3–4 weeks· Prereq: Phases 1–2
▸
Prerequisites

Python fluency (Phase 1) and CLI basics (Phase 0).

What to learn

How the OS manages processes, memory, and files; and how the network works from TCP/IP up through HTTP/HTTPS, REST APIs, DNS, and TLS. This is the substrate every AI backend runs on — you need to reason about it to diagnose failures.

Why it matters

When an LLM API call times out, or your server crashes under load, or a container can't reach the database — you need OS and networking literacy to diagnose, not guess. Every AI engineer operates servers and calls APIs constantly.

Subtopics
Processes & signalsThreads vs processesMemory layout (stack, heap)File system & permissionsEnvironment variablesbash scriptingpipes & grep & awkTCP/IP layersHTTP/HTTPS & TLSDNSREST & JSONcurl & httpieHTTP status codesWebSockets overview
Mini-projects
  • Write bash scripts that process log files with pipes, grep, and awk.
  • Call 5 different public REST APIs from the terminal with curl and parse the JSON with jq.
  • Trace the full path of an HTTP request: DNS → TCP → TLS → HTTP → JSON. Draw it.
Production checkpoint
Production noteAI backends make dozens of external HTTP calls (LLM APIs, vector DBs, cloud storage). Understanding connection pooling, timeouts, retries, and TLS is how you avoid outages.
Common mistakes
AvoidTreating HTTP as magic. Not knowing the difference between a 400, 429, and 503. Being unable to diagnose a connection error from the CLI. These gaps cost hours in production debugging.
Important notes
NoteYou don't need to be a network engineer. The goal is operational literacy: understand enough to read error messages, configure services, and know which layer a problem lives in.
Mastery criteria

You can navigate Linux confidently, write useful bash scripts, explain the HTTP request lifecycle including DNS and TLS, and call and parse a REST API from the terminal.

Trusted resources
The Linux Command Line (Shotts, free)linuxcommand.org MDN — HTTP Overviewdeveloper.mozilla.org MDN — HTTP Status Codesdeveloper.mozilla.org curl Manualcurl.se

Databases & SQL

Intermediate~3–5 weeks· Prereq: Phase 3
▸
Prerequisites

Linux CLI and Python (Phases 1, 3).

What to learn

Relational database design and SQL from schema creation through advanced queries. Covers normalization, joins, indexes, transactions, ACID properties, query explanation, and using an ORM (SQLAlchemy) with migrations (Alembic). PostgreSQL is the standard choice.

Why it matters

Most AI applications store user data, conversation history, evaluation results, and metadata in a relational database. If you can't design schemas or write optimized SQL, you'll ship slow, unreliable services with data integrity bugs.

Subtopics
CREATE / INSERT / SELECT / UPDATE / DELETEJOINs (inner, left, right)GROUP BY / HAVING / ORDER BYSchema design & normalizationPrimary & foreign keysIndexes (B-tree)EXPLAIN ANALYZETransactions & ACIDSQLAlchemy ORMAlembic migrationsConnection pooling
Mini-projects
  • Design a normalized schema for a multi-user chat application with conversation history.
  • Write and optimize 10 complex queries using EXPLAIN ANALYZE to benchmark before/after indexing.
  • Build a CRUD application with SQLAlchemy ORM and Alembic migrations.
Production checkpoint
Production noteNever run schema changes with raw SQL in production. Use migration files (Alembic) that are version-controlled and reversible. A migration that can't be rolled back is a production incident waiting to happen.
Common mistakes
AvoidN+1 query problems (select inside a loop); missing indexes on foreign keys; using SELECT * in production; skipping connection pooling; no transaction boundaries.
Important notes
NoteLearn pgvector at this phase too (overview level) — adding vector similarity search to PostgreSQL is the simplest path to semantic search and you'll use it in AI Phase 15.
Mastery criteria

You can design a normalized schema, write complex multi-join queries, add the right indexes, run migrations, and diagnose a slow query with EXPLAIN ANALYZE.

Trusted resources
PostgreSQL Official Tutorialpostgresql.org SQLAlchemy 2.0 Docssqlalchemy.org Alembic Documentationalembic.sqlalchemy.org PostgreSQL EXPLAINpostgresql.org

Software Design & Patterns

Intermediate~3–4 weeks· Prereq: Phases 1–4
▸
Prerequisites

Solid Python OOP (Phase 1) and hands-on project experience (Phases 2–4).

What to learn

How to structure software that stays maintainable as it grows: SOLID principles, composition over inheritance, dependency injection, and the most useful design patterns (Repository, Factory, Strategy, Observer). Apply these through hands-on refactoring of real code.

Why it matters

AI systems have multiple interchangeable components: swap out an LLM provider, change a vector DB, add a new retrieval strategy. Clean design patterns make these swaps easy. Without them, every change breaks three other things.

Subtopics
SOLID (all 5 principles)Composition vs inheritanceDependency injectionRepository patternFactory patternStrategy patternObserver / event busFacade patternData classes & value objectsRefactoring techniques
Mini-projects
  • Refactor a monolithic script into clean, pattern-based modules; document the before/after.
  • Build a pluggable notification system (Strategy + Observer) with email, Slack, and webhook adapters.
  • Design and implement a Repository layer between a FastAPI service and a database — swap SQLite for PostgreSQL without changing business logic.
Production checkpoint
Production noteThe Repository pattern isolates database logic so tests can use in-memory fakes instead of real DBs. This is the difference between a test suite that runs in 2 seconds and one that takes 2 minutes.
Common mistakes
AvoidOver-engineering small scripts with patterns they don't need. Learning patterns is about recognizing the problem they solve — apply them when you feel the pain, not as a default.
Important notes
NoteDependency injection is the most impactful pattern for testability. Learn to inject dependencies rather than hard-code them — it unlocks clean testing and easy component swapping in AI pipelines.
Mastery criteria

You can explain all 5 SOLID principles with concrete Python examples, apply the Repository pattern to isolate data access, and refactor a class hierarchy to use composition instead of deep inheritance.

Trusted resources
Refactoring Guru — Python Design Patternsrefactoring.guru Python abc (Abstract Base Classes)docs.python.org Dependency Injection in Pythonplainenglish.io

REST APIs & Backend Engineering

Intermediate~4–6 weeks· Prereq: Phases 3–5
▸
Prerequisites

Python (Phase 1), HTTP/REST (Phase 3), databases (Phase 4), design patterns (Phase 5).

What to learn

Build production-quality REST APIs with FastAPI: routing, Pydantic request/response models, authentication (API keys, JWT), middleware, error handling, dependency injection, and connecting to a real database via SQLAlchemy. Write the API you'd be proud to hand off to another engineer.

Why it matters

An LLM call in a notebook is a demo. Turning it into a service with auth, validation, error handling, and a database is the engineering that makes you employable as an AI Engineer. FastAPI is the standard for Python AI backends.

Subtopics
FastAPI routing & path paramsPydantic request/response schemasHTTP methods (GET/POST/PUT/DELETE/PATCH)Query params & paginationDependency injection in FastAPIAPI key authJWT (access + refresh tokens)Middleware (logging, CORS)Global error handlersBackground tasksOpenAPI / Swagger docsDB integration (SQLAlchemy)
Mini-projects
  • Build a full CRUD REST API (e.g. a task manager) with FastAPI, Pydantic, and SQLAlchemy, including JWT auth and proper error codes.
  • Add pagination, filtering, and sorting to a collection endpoint without breaking existing clients.
  • Write a middleware that logs every request with method, path, status, and latency — structured JSON output.
Production checkpoint
Production noteAlways validate inputs with Pydantic — never trust raw request data. Return structured error responses (not stack traces) and use HTTP status codes correctly: 400 for bad input, 401 for unauth, 403 for forbidden, 404 for not found, 422 for validation failure, 500 for server bugs.
Common mistakes
AvoidNo input validation; leaking internal errors in responses; ignoring idempotency (PUT vs POST); storing JWT secrets in code; no rate limiting on public endpoints.
Important notes
NoteFastAPI's dependency injection system is exactly what you'll use to inject LLM clients, vector DB connections, and auth state into route handlers in AI backend work. Learn it deeply now.
Mastery criteria

You can build a secured, validated, database-backed REST API with proper authentication, error handling, logging middleware, and OpenAPI docs — without looking up the FastAPI docs for every step.

Trusted resources
FastAPI Documentationfastapi.tiangolo.com FastAPI Security Tutorialfastapi.tiangolo.com Pydantic Documentationpydantic.dev SQLAlchemy ORM Quickstartsqlalchemy.org

Testing & Code Quality

Intermediate~2–3 weeks· Prereq: Phase 6
▸
Prerequisites

Solid Python (Phase 1) and a real service to test (Phase 6).

What to learn

Write tests that actually catch bugs: unit tests (pytest, fixtures, mocking), integration tests against real infrastructure (TestClient, real DB), coverage measurement, and code quality enforcement (ruff, black, mypy, pre-commit). Understand test pyramid strategy — how many of each type to write.

Why it matters

AI systems are hard to test because LLMs are nondeterministic. The only way to maintain quality is a rigorous offline test suite that runs fast. Engineers who can't write good tests create fear-based codebases where no one dares refactor.

Subtopics
pytest fixtures & parametrizeunittest.mock & MagicMockFastAPI TestClientAsync test supportCoverage (pytest-cov)Integration testsruff (linting)black (formatting)mypy (type checking)pre-commit hooksTest pyramid strategy
Mini-projects
  • Backfill tests for a Phase 6 API until coverage is >80%; document what edge cases you found by writing tests.
  • Set up a pre-commit hook that runs ruff, black, and mypy — reject commits that don't pass.
  • Write an integration test that boots your full FastAPI app (with real DB) and tests an auth-protected endpoint end to end.
Production checkpoint
Production noteTests that pass in isolation but fail together are integration gaps. Always have at least one end-to-end test that proves the whole stack works — it catches config and dependency issues that unit tests miss.
Common mistakes
AvoidOnly testing happy paths; mocking too much (your mocks pass, real code fails); 100% coverage as the goal (coverage measures lines hit, not behavior verified); skipping linting and letting style debt accumulate.
Important notes
NoteIn AI engineering you'll write eval suites (not just unit tests) to measure model quality. The pytest + fixture + parametrize pattern is the same — the skill transfers directly.
Mastery criteria

You can write a test suite with fixtures, mocks, parametrized cases, and integration tests; measure coverage; and enforce code quality automatically via pre-commit and CI.

Concurrency & Async I/O

Advanced~2–3 weeks· Prereq: Phases 1, 6
▸
Prerequisites

Python (Phase 1) and a FastAPI service to apply concurrency patterns to (Phase 6).

What to learn

How Python handles concurrent work: the event loop and asyncio model (I/O bound tasks), threading (I/O with blocking libraries), multiprocessing (CPU-bound work), asyncio.gather, semaphores for rate limiting, and streaming responses. Know when to reach for each and why the GIL matters.

Why it matters

AI backends routinely make concurrent calls to LLM APIs, vector DBs, and external services. An engineer who doesn't understand async programming will either block the event loop (hurting latency) or introduce race conditions.

Subtopics
Event loop & coroutinesasync def / awaitasyncio.gather & create_taskSemaphores for rate limitingThreading (I/O bound)Multiprocessing (CPU bound)concurrent.futuresThread safety & locksStreaming responses (SSE)Background tasks in FastAPIGIL awareness
Mini-projects
  • Build a concurrent API client that calls 10 external APIs in parallel with asyncio.gather and a semaphore to limit concurrency — measure the speedup vs sequential.
  • Add Server-Sent Events streaming to a FastAPI endpoint that yields tokens one by one (simulating LLM streaming).
  • Identify and fix a blocking call in a FastAPI async route that causes the event loop to stall under load.
Production checkpoint
Production noteNever call a blocking I/O function directly in an async route handler — use run_in_executor or an async-native library. One blocked call stalls all requests on that event loop thread.
Common mistakes
AvoidUsing threads when asyncio suffices (overhead); using asyncio for CPU-bound work (use multiprocessing); forgetting to await coroutines; not rate-limiting concurrent API calls (cost explosions).
Important notes
NoteSSE streaming is the main way to stream LLM token output to a browser client. This phase is where that becomes obvious and easy to implement rather than a mystery.
Mastery criteria

You can build a FastAPI service that handles concurrent LLM API calls with proper rate limiting, streams responses via SSE, and offloads blocking work to threads — all without stalling the event loop.

Trusted resources
Python asyncio Documentationdocs.python.org FastAPI — Concurrency & async/awaitfastapi.tiangolo.com MDN — Server-Sent Eventsdeveloper.mozilla.org Python concurrent.futuresdocs.python.org

Docker & Containers

Advanced~2–3 weeks· Prereq: Phases 3, 6
▸
Prerequisites

Linux CLI (Phase 3) and a working FastAPI service to containerize (Phase 6).

What to learn

Containerization with Docker: images, layers, Dockerfiles (multi-stage builds), volumes, networking, and Docker Compose for local multi-service stacks (app + PostgreSQL + Redis). Learn to build lean, reproducible production images and troubleshoot container issues.

Why it matters

"Works on my machine" is unacceptable in production. Containers ensure your app runs identically in development, CI, staging, and production. Every AI service in the industry is containerized.

Subtopics
Images vs containersDockerfile (FROM, RUN, COPY, CMD, ENTRYPOINT)Multi-stage buildsLayer cachingVolumes & bind mountsContainer networkingDocker ComposeEnvironment variables & secrets.dockerignoreHealth checksContainer registries
Mini-projects
  • Write a production Dockerfile for a FastAPI app using a multi-stage build — final image under 150MB, non-root user, health check included.
  • Create a Docker Compose stack: FastAPI app + PostgreSQL + Redis + Adminer for local development. One command to boot the whole stack.
  • Push the image to GitHub Container Registry (GHCR) and pull it on a different machine to verify reproducibility.
Production checkpoint
Production noteNever run containers as root. Always add a health check endpoint (/health) and configure it in the Dockerfile — orchestrators use it to know when to kill and restart containers.
Common mistakes
AvoidGiant single-layer images with dev dependencies included; secrets baked into image layers (they're visible in history); no .dockerignore (copies .git and node_modules); running as root.
Important notes
NoteMulti-stage builds are the standard for production Python containers: one stage to install deps, a second lean stage to copy only what's needed. This halves image size and removes build tools from production.
Mastery criteria

You can write a production-grade multi-stage Dockerfile, boot a full local stack with Compose, push/pull images from a registry, and explain how layers, volumes, and container networking work.

Trusted resources
Docker Getting Starteddocs.docker.com Docker Development Best Practicesdocs.docker.com Docker Compose Documentationdocs.docker.com GitHub Container Registry (GHCR)docs.github.com

CI/CD & DevOps Basics

Advanced~2–3 weeks· Prereq: Phases 7, 9
▸
Prerequisites

Testing (Phase 7) and Docker (Phase 9) — CI/CD automates both.

What to learn

Automate quality gates and deployment: GitHub Actions workflows for lint, type check, test, build, and deploy. Understand when CI fails vs passes and why. Practice deployment strategies: rolling updates, rollback, and feature flags at a basic level.

Why it matters

Manual deployments are error-prone and slow. CI/CD enforces quality (tests must pass before merge), speeds up delivery, and makes rollback fast when something breaks. Every AI project you deploy needs CI — evaluations, model registration, and service deploys should all be automated.

Subtopics
GitHub Actions workflows (YAML)Triggers (push, PR, schedule)Jobs & stepsSecrets & environment variablesMatrix builds (multiple Python versions)Build & push Docker imageDeploy on merge to mainBranch protection rulesRelease tags & versioningBasic deployment strategies
Mini-projects
  • Create a GitHub Actions workflow that runs ruff, mypy, and pytest on every push and PR — block merge if any fails.
  • Add a second job that builds and pushes a Docker image to GHCR on merge to main.
  • Add a scheduled workflow that runs your test suite weekly against the latest dependencies and alerts on failure.
Production checkpoint
Production noteNever merge secrets into CI YAML — use GitHub Actions encrypted secrets. CI workflows are checked into the repo and are readable by anyone with repo access.
Common mistakes
AvoidNo branch protection rules (anyone can push directly to main); long-running CI that discourages developers; hardcoding credentials in YAML files; no automated tests (CI with no tests is just slow lint).
Important notes
NoteIn AI engineering, CI runs your eval suite on model or prompt changes. The pattern is identical: a GitHub Actions workflow that runs on PR and blocks merge if evaluation metrics regress.
Mastery criteria

You can write a GitHub Actions workflow that lints, type-checks, tests, builds, and deploys a Dockerized service — with secrets managed correctly and branch protection enforcing all gates.

Trusted resources

System Design Fundamentals

Advanced~3–4 weeks· Prereq: Phases 4, 6, 9
▸
Prerequisites

Databases (Phase 4), REST backend (Phase 6), and containers (Phase 9).

What to learn

How to think about and design distributed systems under real constraints: components, data flow, storage choices, caching strategies, queues, load balancing, consistency vs availability trade-offs, and back-of-envelope capacity estimation. Document decisions with ADRs.

Why it matters

Senior AI engineers are hired to make trade-offs, not just call APIs. System design skills are what let you architect a production RAG system, a model-serving platform, or a multi-agent orchestration service that survives real traffic.

Subtopics
System components (LB, cache, queue, DB)Vertical vs horizontal scalingCaching (CDN, Redis, in-process)Message queues (overview)Load balancing strategiesCAP theoremConsistency modelsBack-of-envelope estimationAPI design decisionsArchitecture Decision RecordsDiagrams (C4 model)
Mini-projects
  • Design a URL shortener end to end: capacity estimate, schema, caching, redirect API, and failure modes — document it as a design doc.
  • Write 3 Architecture Decision Records (ADRs) for real decisions made in your Phase 6–9 projects.
  • Draw a C4 context + container diagram for your most complex project so far.
Production checkpoint
Production noteDesign for failure: every external dependency will eventually be slow or down. Always define what happens when your database, LLM API, or vector store is unavailable — fallback behavior is not optional.
Common mistakes
AvoidDesigning only the happy path; no capacity estimates; adding caching without measuring cache hit rate; over-engineering a monolith into microservices before the system warrants it.
Important notes
NoteAI system design is classical system design with extra constraints: token budgets, model latency variance, retrieval pipelines, and cost control. The classics (DDIA, System Design Primer) are timeless — AI specifics layer on top.
Mastery criteria

You can whiteboard a medium-complexity system (URL shortener, chat app, file store), give a capacity estimate, identify failure modes, and document your decisions with ADRs.

Trusted resources
System Design Primergithub.com Designing Data-Intensive Applicationsdataintensive.net C4 Model for Architecture Diagramsc4model.com Architecture Decision Records (ADR)adr.github.io

04 · Prove It

Stress Tests

Each stress test is a hard, open-ended challenge designed to surface the difference between tutorial knowledge and real understanding. Pass all of these before moving to AI Engineering phases. These are the gates.

STRESS TEST 01 · Phase 1Beginner

Cold-Start Python Package

~3–5 hrs
Challenge
From a blank terminal, create a typed, packaged Python library from scratch — no templates. Must include: abstract base class, two implementations, full type hints, pytest suite, pyproject.toml, and pass mypy . with strict mode.
Gate
You cannot Google "how to set up a Python package." Everything must come from memory or the official docs.
Pass criteria
pip install . succeeds; pytest passes; mypy --strict . exits 0; no global pip installs.
STRESS TEST 02 · Phase 1Beginner

Debug the Async Bug

~2–4 hrs
Challenge
A FastAPI async endpoint that calls 5 external APIs appears to take 10 seconds instead of 2. You are given the broken code. Diagnose the root cause, explain why it's slow (blocking vs non-blocking), and fix it.
Gate
Fix must actually be async-concurrent — wrapping in a thread is not a valid solution.
Pass criteria
Fixed version completes all 5 API calls in <max_latency_of_one + 200ms; root cause explained in writing.
STRESS TEST 03 · Phase 2Beginner

Implement a Graph Traversal

~2–3 hrs
Challenge
Given an unweighted directed graph represented as an adjacency list, implement both BFS and DFS from scratch without looking up the algorithms. Find shortest path (BFS) and detect cycles (DFS). Full tests required.
Gate
No pseudocode lookup. Explanation of time and space complexity written in a docstring.
Pass criteria
All test cases pass including disconnected graphs, self-loops, and the empty graph.
STRESS TEST 04 · Phase 3Beginner

Diagnose a Network Failure

~1–2 hrs
Challenge
Given a broken Docker Compose stack where the API can't reach the database, diagnose and fix it using only CLI tools (no GUI). Document: what you ran, what you saw, and what the root cause was.
Gate
You must use docker logs, curl, nc or telnet, and container exec to diagnose — no changing random settings until it works.
Pass criteria
Root cause correctly identified in writing; fix applied with one targeted change; explanation of what was wrong and why.
STRESS TEST 05 · Phase 4Intermediate

Optimize a Slow Query

~2–4 hrs
Challenge
You are given a schema with 1M rows and a query that takes 8 seconds. Using only EXPLAIN ANALYZE, add indexes to bring it under 50ms. Write down every step of your diagnosis before touching any index.
Gate
No ORM — raw SQL only. Must explain what the query planner was doing and why the index helps.
Pass criteria
Query runs in <50ms; written explanation of the sequential scan and why the B-tree index fixed it.
STRESS TEST 06 · Phase 4Intermediate

Fix a Migration Disaster

~2–3 hrs
Challenge
A migration was run in production that dropped a required column. You are given the broken state. Write a rollback migration, apply it, and explain what process controls would have prevented this.
Gate
Rollback migration must not lose existing data. Process controls must be specific (not just "be careful").
Pass criteria
Rollback migration works on a real database; 3+ specific process controls documented.
STRESS TEST 07 · Phase 5Intermediate

Refactor to Open/Closed

~3–5 hrs
Challenge
You are given a PaymentProcessor class with a giant if/elif chain for Visa, Mastercard, and PayPal. Refactor it so adding a new payment method touches zero existing code. Write tests for each step.
Gate

Pass criteria
Refactor complete; adding "Crypto" payment requires zero changes to existing files; all tests green.
STRESS TEST 08 · Phase 6Intermediate

Abuse a Real API

~2–4 hrs

Given a running FastAPI service, abuse it: send malformed JSON, SQL injection strings, oversized payloads, missing auth headers, and concurrent duplicate POST requests. Document every failure mode you find. Then fix them all.
Gate
Every failure must be reproduced by a test before the fix. Fixes must be targeted (not just "validate everything").

STRESS TEST 09 · Phase 7Intermediate
