Orientation & Setup

Beginner~1 week· Prereq: none
▸
Prerequisites

None. A computer and consistent weekly study time.

What to learn

Set up a professional environment: a code editor, Python via a version manager, isolated virtual environments, the terminal, and Git/GitHub. Build a personal learning system (notes, spaced repetition, a project log).

Why it matters

Most beginners lose weeks to broken environments and lost work. A clean, reproducible setup and a habit of version control are the substrate every later phase depends on.

Subtopics
VS CodePython install (pyenv / official)venv & pipconda (optional)Terminal basicsGit init / commit / branchGitHub repos & SSHMarkdownJupyter / notebooks
Mini-projects
  • Create a GitHub repo, write a README in Markdown, and push your first commit.
  • Set up a virtual environment and pin dependencies in requirements.txt.
  • Start a public "learning log" repo you will update through the whole roadmap.
Production checkpoint
Production noteReproducibility starts now: every project gets its own isolated environment and a committed dependency lockfile. "Works on my machine" is a defect, not an excuse.
Common mistakes
AvoidInstalling everything into the global Python; never using branches; committing secrets or large data files. Add a .gitignore on day one.
Important notes
NoteDon't over-optimize tooling. Pick one editor and one environment tool and move on — tool-tweaking is a common form of procrastination.
Mastery criteria

You can create an isolated environment, install packages, and push branched, ignored-secret code to GitHub without looking up the steps.

Computer Science Foundations

Beginner~3–5 weeks· Prereq: Phase 0
▸
Prerequisites

Phase 0 environment and Git.

What to learn

How computers represent and process data; core data structures and algorithms; algorithmic complexity (Big-O); the Linux command line; and how the web works (HTTP, JSON, clients/servers).

Why it matters

AI systems are software systems. Without complexity intuition you write code that doesn't scale; without CLI and HTTP fluency you can't operate servers, call APIs, or debug deployments.

Subtopics
Binary, memory, encodingArrays, hash maps, trees, graphsSorting & searchingRecursionBig-O time/spaceLinux files & processesbash, pipes, grepHTTP, REST, JSON, status codesClient/server model
Mini-projects
  • Implement a hash map and a binary search from scratch; analyze their Big-O.
  • Write a bash script that processes log files with pipes and grep.
  • Call a public REST API from the terminal with curl and parse the JSON.
Production checkpoint
Production noteIn real systems, the difference between O(n) and O(n²) is the difference between a 200ms and a 20-minute job. Always estimate cost before scaling data up.
Common mistakes
AvoidGrinding hundreds of competitive-programming puzzles. You need working fluency, not contest mastery — learn enough to reason about cost and pick the right structure.
Important notes
Skip for nowAdvanced algorithm design (DP-heavy interview grinds, complex graph algorithms) can wait until Phase 27 interview prep.
Mastery criteria

You can state the Big-O of common operations, navigate Linux confidently, and explain a request's path from client to server and back.

Python Programming for AI

Beginner~4–6 weeks· Prereq: Phases 0–1
▸
Prerequisites

Phase 1 CS basics and the CLI.

What to learn

Idiomatic Python end-to-end: data types, control flow, functions, OOP, modules/packages, error handling, type hints, virtual environments, dependency management, testing with pytest, and async I/O. This is the most important foundation for an AI Engineer.

Why it matters

Nearly all AI tooling, APIs, and ML frameworks are Python-first. Strong, testable, well-structured Python is what separates someone who pastes snippets from someone who ships systems.

Subtopics
Lists/dicts/sets/tuplesComprehensionsFunctions & *args/**kwargsClasses & dunder methodsModules & packagesExceptions & context managersType hints + mypyGenerators & iteratorsDecoratorsasync / awaitpytestrequests / httpxpydantic
Mini-projects
  • Build a CLI tool (e.g., a file organizer or web-API client) with argument parsing and tests.
  • Write a small library with type hints, packaged with pyproject.toml and a pytest suite.
  • Fetch and validate data from a public API using httpx + pydantic.
Production checkpoint
Production noteType hints + pydantic validation catch entire classes of bugs before runtime. Treat untyped, untested Python as a liability in any service you intend to deploy.
Common mistakes
AvoidJumping to ML libraries before you can write a clean class or a test. Shallow Python is the #1 reason later phases feel impossible.
Important notes
NoteLearn async early enough that LLM streaming and concurrent API calls (Phases 12, 19) aren't a wall. You don't need deep concurrency theory yet.
Mastery criteria

You can design a small package with classes, type hints, and tests, handle errors gracefully, and call APIs both synchronously and asynchronously.

Math for AI

Beginner~4–8 weeks· Prereq: Phase 2 (parallel ok)
▸
Prerequisites

Comfort with Python (Phase 2) helps you implement what you learn.

What to learn

The applied math behind ML: linear algebra (vectors, matrices, dot products), calculus (derivatives, gradients, the chain rule), and the basics of optimization (gradient descent). Focus on intuition and computation, not proofs.

Why it matters

Embeddings are vectors, neural networks are chains of matrix multiplications, and training is optimization. Without this intuition, deep learning stays a black box you can't debug.

Subtopics
Vectors & normsMatrix multiplicationDot product & cosine similarityEigenvalues (intuition)Derivatives & partialsChain ruleGradientsLoss surfacesGradient descentVectorized NumPy
Mini-projects
  • Implement linear regression with gradient descent in NumPy from scratch.
  • Compute cosine similarity between word vectors and rank nearest neighbors.
  • Visualize a 2D loss surface and the path gradient descent takes.
Production checkpoint
Production noteCosine similarity and vector norms reappear directly in embeddings and RAG (Phases 14–15). The math here is not academic — it's the engine of semantic search.
Common mistakes
AvoidTrying to master pure-math rigor (epsilon-delta proofs, abstract algebra). You need working intuition for ML, not a math degree.
Important notes
Skip for nowMeasure theory, advanced multivariate calculus, and convex-optimization theory are master-level (Phase 28) topics for research tracks only.
Mastery criteria

You can derive and implement gradient descent for linear regression and explain why gradients point toward steeper loss.

Data Handling and Analysis

Beginner~3–5 weeks· Prereq: Phases 2–3
▸
Prerequisites

Python (Phase 2) and basic vector math (Phase 3).

What to learn

Acquire, clean, transform, query, and visualize real-world data with NumPy, Pandas, Matplotlib, and SQL. Learn to do disciplined exploratory data analysis (EDA) and to spot data quality problems.

Why it matters

In practice, AI projects are 60–80% data work. Models are only as good as the data feeding them; messy or leaking data sinks every downstream phase.

Subtopics
NumPy arrays & broadcastingPandas DataFramesIndexing, joins, groupbyMissing values & typesOutliers & duplicatesMatplotlib chartsSQL SELECT/JOIN/GROUP BYCSV / ParquetEDA workflow
Mini-projects
  • Clean a messy public dataset (missing values, types, duplicates) and document each decision.
  • Produce an EDA notebook with distributions, correlations, and clear takeaways.
  • Load data into PostgreSQL and answer 10 business questions with SQL.
Production checkpoint
Production noteReal pipelines validate schemas and ranges automatically. Manual cleaning in a notebook doesn't survive contact with changing production data — automate and assert.
Common mistakes
AvoidDrawing conclusions before checking data quality; ignoring class imbalance; deleting rows with missing values without understanding why they're missing.
Important notes
NoteLearn SQL alongside Pandas — most production data lives in databases, not CSV files. They reinforce each other.
Mastery criteria

You can take a raw, messy dataset to a clean, well-understood, documented state and query it fluently in both Pandas and SQL.

Statistics and Probability

Intermediate~3–5 weeks· Prereq: Phases 3–4
▸
Prerequisites

Math intuition (Phase 3) and data skills (Phase 4).

What to learn

Descriptive and inferential statistics, probability distributions, sampling, hypothesis testing, confidence intervals, Bayes' theorem, and the distinction between correlation and causation.

Why it matters

Evaluation, A/B testing, uncertainty estimates, and reasoning about model performance all rest on statistics. Misreading a metric or a test is how teams ship the wrong model confidently.

Subtopics
Mean/variance/stdDistributions (normal, binomial)Central Limit TheoremSampling & biasHypothesis tests & p-valuesConfidence intervalsBayes' theoremCorrelation vs causationA/B testing
Mini-projects
  • Simulate the Central Limit Theorem and bootstrap a confidence interval.
  • Design and analyze a simulated A/B test, including statistical significance.
  • Apply Bayes' theorem to a realistic diagnostic-test problem.
Production checkpoint
Production noteOnline model comparisons are A/B tests. If you can't compute and interpret significance, you can't safely tell whether a new model actually beat the old one.
Common mistakes
AvoidTreating p-values as truth, p-hacking, confusing correlation with causation, and ignoring base rates (a classic Bayes trap).
Important notes
NoteYou need applied statistical literacy, not theorem-proving. Focus on the mistakes that cause real-world misjudgments.
Mastery criteria

You can design a sound experiment, interpret its results honestly with uncertainty, and explain base-rate reasoning.

Machine Learning Foundations

Intermediate~6–8 weeks· Prereq: Phases 4–5
▸
Prerequisites

Data (Phase 4), statistics (Phase 5), and math intuition (Phase 3).

What to learn

The classical ML toolkit and, more importantly, the discipline of doing ML correctly: supervised/unsupervised learning, the bias–variance tradeoff, regularization, proper train/validation/test splits, cross-validation, feature engineering, and metric selection — all in scikit-learn.

Why it matters

Most real business problems are still solved best by classical ML, not LLMs. The methodology you build here (no leakage, honest evaluation) is the bedrock of every model you'll ever ship.

Subtopics
Linear & logistic regressionDecision treesRandom forestsGradient boosting (XGBoost/LightGBM)k-NN, SVMk-means & PCABias–variance & overfittingRegularizationCross-validationFeature engineeringMetrics: ROC-AUC, F1, RMSEData leakage
Mini-projects
  • Build a classification model with a full scikit-learn Pipeline (no leakage) and proper cross-validation.
  • Compare a linear model, a tree, and gradient boosting on the same task with honest metrics.
  • Cluster customers with k-means and reduce dimensions with PCA for visualization.
Production checkpoint
Production noteFit preprocessing only on training folds, inside the pipeline. Leakage inflates offline scores and silently destroys production accuracy — it is the most common and most expensive ML bug.
Common mistakes
AvoidOptimizing accuracy on imbalanced data; tuning on the test set; data leakage via global scaling/imputation; ignoring a baseline.
Important notes
NoteAlways start with a dumb baseline (majority class, mean predictor). If your model can't beat it, the problem is your data or framing, not the algorithm.
Mastery criteria

You can frame a problem, build a leakage-free pipeline, choose the right metric, and explain why your validation strategy gives an honest estimate.

Practical ML Engineering

Intermediate~3–5 weeks· Prereq: Phase 6
▸
Prerequisites

Classical ML methodology (Phase 6) and solid Python (Phase 2).

What to learn

Turn notebook experiments into reproducible, modular training pipelines: clean project structure, configuration management, deterministic seeds, data/feature versioning, and saved model artifacts. This is the bridge from "data science" to "ML engineering".

Why it matters

A model that can't be retrained reproducibly is a dead end. Engineering discipline here is what makes Phases 20–25 (MLOps, deployment, system design) possible.

Subtopics
Project structure (src/ layout)Config managementDeterministic seedsTrain/eval scripts (not just notebooks)Data versioning (DVC)Artifact savingFeature pipelinesLoggingMakefiles / task runners
Mini-projects
  • Refactor a notebook into a CLI training pipeline with a config file and saved artifacts.
  • Version a dataset with DVC and reproduce a run from a clean checkout.
  • Write an evaluation script that loads the saved model and reports metrics on a held-out set.
Production checkpoint
Production note"Reproducible" means a teammate can clone, run one command, and get your numbers. Pin seeds, versions, and data hashes — otherwise debugging a regression is impossible.
Common mistakes
AvoidKeeping everything in one giant notebook; hardcoding paths and hyperparameters; never versioning data; relying on cell-execution order.
Important notes
NoteNotebooks are great for exploration, terrible for pipelines. Learn to graduate code out of notebooks into tested modules.
Mastery criteria

You can take an experiment and produce a config-driven, version-controlled, reproducible training + evaluation pipeline.

Deep Learning Foundations

Intermediate~6–10 weeks· Prereq: Phases 3, 6
▸
Prerequisites

Calculus/linear-algebra intuition (Phase 3) and ML methodology (Phase 6).

What to learn

How neural networks work and how to train them in PyTorch: tensors, autograd, the training loop, activation and loss functions, optimizers, regularization, and using a GPU. Build a small net from scratch before relying on high-level abstractions.

Why it matters

Transformers and LLMs (Phases 11–12) are deep networks. Understanding backprop, optimization, and PyTorch internals is what lets you debug training, fine-tune models, and reason about behavior.

Subtopics
Tensors & autogradForward/backward passBackpropagationActivations (ReLU, GELU)Loss functionsOptimizers (SGD, Adam)Learning-rate schedulesDropout & batch normDatasets & DataLoadersGPU trainingCNN & RNN basics
Mini-projects
  • Implement a 2-layer neural net and backprop by hand in NumPy, then reproduce it in PyTorch.
  • Write a full PyTorch training loop (Dataset, DataLoader, optimizer, validation) for an image or tabular task.
  • Diagnose and fix an overfitting model using regularization and early stopping.
Production checkpoint

Important notes

Mastery criteria

You can write a complete PyTorch training loop from scratch, explain backprop, and debug a model that won't converge.

NLP Basics

Intermediate~3–5 weeks· Prereq: Phases 6, 8
▸
Prerequisites

ML methodology (Phase 6) and deep learning (Phase 8).

What to learn

Classic text processing and representations: tokenization, normalization, TF-IDF, word embeddings (word2vec/GloVe), and core tasks like text classification, sentiment, and named-entity recognition.

Why it matters

Understanding tokenization and embeddings is essential before transformers and LLMs. It also makes the limits and costs of modern models intuitive (e.g., why token count drives price and latency).

Transformers & Foundation Models

Advanced~4–6 weeks· Prereq: Phases 8, 10
▸
Prerequisites

Deep learning (Phase 8) and NLP basics (Phase 10).

What to learn

The transformer architecture in depth: self-attention, multi-head attention, positional encodings, encoder vs decoder stacks, and the pretraining paradigm that underlies BERT, GPT, and modern foundation models. Use the Hugging Face Transformers library hands-on.

Why it matters

Transformers are the foundation of every modern LLM. Real understanding here is what separates an AI Engineer from a prompt-only user — it's how you reason about context windows, attention costs, and model behavior.

Subtopics
Self-attention (Q, K, V)Scaled dot-product attentionMulti-head attentionPositional encodingEncoder vs decoderResiduals & layer normPretraining objectivesBERT vs GPT familiesTransformers library & Hub
Mini-projects
  • Read "Attention Is All You Need" and write a plain-English explanation of attention.
  • Implement scaled dot-product attention in PyTorch from scratch.
  • Load a pretrained model from the Hub and fine-tune it on a small classification task.
Production checkpoint
Production noteAttention is roughly quadratic in sequence length — long contexts cost more memory, latency, and money. This constraint shapes RAG and chunking decisions in Phases 14–15.
Common mistakes
AvoidTreating transformers as magic; skipping the attention mechanism; assuming bigger context is always better and free.
Important notes
NoteYou don't need to train a large model from scratch. The goal is deep conceptual understanding plus practical use of pretrained models.
Mastery criteria

You can explain attention end-to-end, implement it, and articulate why context length affects cost and latency.

Large Language Models

Advanced~4–6 weeks· Prereq: Phase 11
▸
Prerequisites

Transformers (Phase 11) and async Python (Phase 2).

What to learn

How to actually use LLMs: API usage (OpenAI, Anthropic), tokens and context windows, decoding parameters (temperature, top-p), streaming, structured outputs and tool/function calling, plus running open models locally. Understand inherent limits: hallucination, knowledge cutoffs, and nondeterminism.

Why it matters

This is the core of modern AI engineering. Everything later — prompts, RAG, agents, fine-tuning — builds on fluent, cost-aware LLM API usage and a realistic mental model of what these models can and cannot do.

Subtopics
LLM API calls & authTokens & context windowsSystem / user / assistant rolesTemperature & top-pStreaming responsesStructured output (JSON)Tool / function callingLocal models (Ollama, vLLM)Cost & latencyHallucination & limits
Mini-projects
  • Build a streaming chat app over an LLM API with token usage and cost tracking.
  • Force structured JSON output and validate it with pydantic; handle malformed responses.
  • Run an open model locally with Ollama and compare quality, latency, and cost to a hosted API.
Production checkpoint
Production noteAlways set timeouts, retries with backoff, and token/cost limits on LLM calls. Never store API keys in code — use environment variables or a secrets manager (see Phase 23).
Common mistakes
AvoidTrusting outputs blindly; ignoring token costs; hardcoding API keys; assuming determinism; sending sensitive data to third-party APIs without a privacy review.
Important notes
NotePrompt engineering (Phase 13) and RAG (Phase 14) build directly on this. Get streaming, structured output, and tool calling solid first.
Mastery criteria

You can build a robust, streaming, cost-tracked LLM application with structured outputs and graceful error handling.

Prompt Engineering

Advanced~2–3 weeks· Prereq: Phase 12
▸
Prerequisites

Fluent LLM API usage (Phase 12).

What to learn

Reliable prompting techniques: clear instructions, zero/few-shot examples, chain-of-thought, output formatting, role/system design, and — crucially — treating prompts as versioned, tested artifacts rather than throwaway strings.

Why it matters

Subtopics

03 · How Deep

Learning Depth Levels

Not every topic needs the same depth. For each skill, decide the target level and stop over-investing in areas that only need awareness. Use these four levels to calibrate every phase above.

Level 1

Awareness

You know the concept exists, what problem it solves, and roughly when it applies. You can follow a conversation and look things up. Goal: recognize the term and decide whether to go deeper. Fine for niche or future topics (e.g. distributed training as a beginner).

Level 2

Working Knowledge

You can use the tool or technique for a typical task by following docs, debug common errors, and adapt examples. Goal: get things done with reference material. This is the right target for most supporting tools in your stack.

Level 3

Production Level

You can use it reliably in a real system: handle edge cases, errors, security, performance, and cost, and you know its failure modes. Goal: ship it to users with confidence. Target this for your core skills — Python, LLM APIs, RAG, backend, deployment.

Level 4

Master Level

You understand the internals, can teach it, make non-obvious trade-offs, and read primary sources to push past the docs. Goal: design new solutions and lead decisions. Reserve this for your chosen specialization, not everything.

07 · Read First

Critical Notes Before You Study

The fastest way to waste months is to study in the wrong order or skip the unglamorous parts. Internalize these before you start.

Order mattersDo not jump straight into LLM agents. Without Python, APIs, and data fundamentals, agent frameworks are magic you cannot debug.
Build, don't just watchTutorials create the illusion of progress. You only learn by building projects that break and fixing them yourself.
Avoid course hoardingCollecting twenty courses is procrastination. Pick one per topic, finish it, and build something with it.
Never ignore evaluationIf you cannot measure quality, you cannot improve it. Evaluation is a first-class skill, not an afterthought.
Control cost earlyLLM and GPU bills scale silently. Track tokens and spend from your very first deployed project.
Never expose API keysKeys in code or the frontend get scraped and abused within hours. Use environment variables and secret stores.
Don't deploy nakedNo AI app should ship without logging, monitoring, rate limits, and safety checks. A demo is not a product.
Don't trust outputs blindlyModels hallucinate confidently. Validate, ground with retrieval, and never let raw output trigger privileged actions.
Prompting ≠ engineeringPrompt engineering is one skill. Full AI engineering is data, retrieval, backend, eval, deployment, and ops.
Fundamentals before frameworksReaching for a framework before understanding the underlying concept leaves you helpless when it breaks.
08 · Avoid These

Common Mistakes

Patterns that repeatedly stall beginner AI engineers. For each: what it is, why it is dangerous, how to avoid it, and the correct behavior.

01 Skipping fundamentals for hype

WhatDiving into LLMs/agents without Python, data, or ML basics.

Why dangerousYou can copy code but cannot debug, evaluate, or extend it — progress stalls at the first real problem.

How to avoidFollow the phase order; do not start Phase 12+ until Phases 2–7 are solid.

Correct behaviorBuild foundations first; the advanced topics then take days, not months.

02 Tutorial loop with no projects

WhatEndlessly watching courses without building anything original.

Why dangerousPassive watching does not build real skill or anything you can show an employer.

How to avoidAfter each topic, build a small project from a blank file without following along.

Correct behaviorTreat tutorials as references, not as the work itself.

03 No evaluation

WhatShipping models or LLM features based on a few cherry-picked examples.

Why dangerousYou have no idea if changes help or hurt; quality silently regresses.

How to avoidBuild an eval set early; measure every change against it.

Correct behaviorTreat evaluation as part of every feature, in CI where possible.

04 Ignoring cost and latency

WhatOptimizing only for quality, never measuring spend or response time.

Why dangerousProducts become unaffordable or too slow to use, and the bill arrives as a shock.

How to avoidTrack tokens, cost per request, and p95 latency from day one.

Correct behaviorBalance cost, latency, and quality as one trade-off.

05 Leaking secrets

WhatHard-coding API keys or committing them to Git or the frontend.

Why dangerousBots scrape public repos within minutes; abuse and huge bills follow.

How to avoidUse environment variables and a secrets manager; never expose keys client-side.

Correct behaviorRoute all model calls through your backend with server-side keys.

06 Trusting model output blindly

WhatTreating generated text as fact or letting it trigger actions unchecked.

Why dangerousHallucinations and prompt injection cause wrong answers and security holes.

How to avoidGround answers with retrieval, validate outputs, and require confirmation for actions.

Correct behaviorTreat model output as an untrusted suggestion, not a command.

07 Over-relying on frameworks

WhatUsing high-level frameworks without understanding what they do underneath.

Why dangerousWhen the abstraction leaks, you cannot debug or optimize it.

How to avoidBuild a basic version by hand once (e.g. a raw RAG loop) before adopting a framework.

Correct behaviorUse frameworks to save time, not to replace understanding.

08 Deploying without ops

WhatShipping with no logging, monitoring, rate limits, or fallbacks.

Why dangerousFailures are invisible, costs run unchecked, and one bad upstream call takes everything down.

How to avoidAdd observability and limits before launch, not after the first incident.

Correct behaviorTreat monitoring, limits, and fallbacks as launch requirements.
